PROJECT BACKGROUND

项目背景

青少年心理健康现状
  • 12-18岁群体心理问题日益突出，焦虑抑郁比例上升
  • 专业心理咨询师资源匮乏，师生比严重不足
  • 学生对传统心理咨询存在病耻感和抵触心理
  • 早期情绪干预缺失，问题往往积累至危机才被发现
核心痛点分析
  • 求助门槛高：面对面咨询心理压力大，预约流程复杂
  • 隐私顾虑：担心个人信息泄露，不愿主动求助
  • 家校信息不对称：家长/教师无法及时了解学生情绪状态
  • 缺乏持续性支持：传统咨询受限于时间和地点
项目定位：AI 情绪成长伙伴 —— 零压力的第一心理求助站

「你的情绪星球，永远亮着灯」

  • 状态管理：Riverpod
  • 加密：AES-256-GCM + SQLCipher
  • 实时通信：WebSocket
  • 本地数据库：sqflite_sqlcipher
  • 后台任务：Workmanager
  • 密钥管理：Flutter Secure Storage
双前端架构选型理由
  • Web 前端：快速开发、跨平台访问、便于家长端轻量化使用，无需安装 App
  • 原生前端：保障数据本地加密存储、离线可用性、推送通知，适合学生日常高频使用
  • 两者共用后端 API，通过角色权限区分功能，实现代码复用与体验优化

CHALLENGES & SOLUTIONS

技术难点与解决方案

难点 1：AI 对话安全边界

挑战：AI生成内容可能包含不当建议或忽略风险信号

解决方案：

双层风险检测（L1关键词+L2语义分析）+ 多层内容审核 + CBT框架约束 + 危机热线自动推送

难点 2：未成年人数据隐私

挑战：未成年人个人信息保护要求极高，数据泄露风险大

解决方案：

AES-256-GCM端到端加密 + SQLCipher本地加密 + 数据最小化采集 + 匿名注册 + 一键删除

难点 3：多端状态同步

挑战：Flutter原生与React Web需实时同步对话与情绪状态

解决方案：

WebSocket实时通信 + Zustand状态管理 + REST API数据同步 + Redis会话缓存

难点 4：后端技术栈迁移

挑战：从Python AI + Go API迁移至Java统一服务层

解决方案：

渐进式迁移策略：先迁移API层至Spring Boot，再整合AI服务为Java服务层，保留Python引擎接口

ENGINEERING CHALLENGES

工程角度难点分析

1. 跨平台一致性 — Flutter 与 React 双前端维护

两套前端代码库需要保持功能一致性和UI统一性，增加了维护成本。通过共用后端API、统一设计规范和类型定义（TypeScript类型共享），降低不一致风险。

统一API契约 + 共享类型定义 + 设计系统规范化

2. AI 模型选型与微调 — Qwen/DeepSeek 本地化 vs API 调用

本地部署需GPU资源投入但延迟低、数据不出境；API调用成本低但依赖外部服务。当前MVP阶段采用API调用快速验证，后续计划逐步迁移至本地微调模型。

渐进式策略：API验证 → 本地微调 → 混合部署

3. 消息队列可靠性 — Kafka/RabbitMQ 高并发对话场景

在高并发AI对话场景下，消息队列需保障消息不丢失、顺序性和 Exactly-Once 语义。通过消息持久化、消费者确认机制和死信队列处理异常消息。

消息持久化 + ACK确认 + 死信队列 + 监控告警
