本索引文件用于对星屿 StarIsle 应用程序的安全评估与供应链安全材料进行版本归档管理，确保：
1. **可追溯性**：每份文档的版本、提交SHA、存储位置可追溯。

4. **可查阅性**：统一索引便于快速定位与查阅各份评估材料。

### 6.2 变更流程

1. **修改申请**：安全团队提出文档修改需求。
2. **修改评审**：技术负责人评审修改内容。
3. **版本递增**：次版本号递增（如 v1.0 → v1.1）；重大变更递增主版本号（如 v1.x → v2.0）。
4. **提交归档**：通过 Git commit 提交，commit message 遵循 `docs(security): <变更说明>` 规范。
5. **索引更新**：同步更新本归档索引的提交SHA与版本号。
6. **复审触发**：每季度或重大架构变更时触发 ATT&CK 评估复审。

## 7. 重要说明

> **关于 MITRE ATT&CK 评估的说明**：
>
> MITRE ATT&CK Evaluations (ER8) 官方注册已于 2026年4月30日 截止，且该评估主要面向安全产品厂商（EDR/XDR/SIEM等）。星屿 StarIsle 作为应用程序，本次评估采用 **MITRE ATT&CK 框架作为威胁建模和安全评估的方法论工具**，对应用自身的安全态势进行系统性评估，而非参加 MITRE 官方评估认证。
>
> 评估结果将作为：
> 1. 内部安全改进的依据（详见04号文档第5章改进建议）；
> 2. 第三方安全审计的参考材料；
> 3. 合规审计（PIPL/未保法）的证据链组成部分。

> **关于 SLSA 构建来源证明的说明**：
>
> SLSA（Supply-chain Levels for Software Artifacts）是一套软件供应链安全框架。通过配置 `actions/attest-build-provenance` GitHub Action，项目自动达到 **SLSA Build Track Level 2**。构建来源证明由 Sigstore 短期证书签名，存储于 GitHub Attestations API，可通过 `gh attestation verify` 命令验证。
>
> SLSA 证明的价值：
> 1. 确保构建产物可追溯到具体源代码提交；
> 2. 通过密码学签名验证产物完整性；
> 3. 满足供应链安全合规要求；
> 4. 防止软件供应链攻击（如构建过程篡改）。
