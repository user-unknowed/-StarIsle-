## 1. 安全架构总览

### 1.1 系统概览

星屿 StarIsle 是面向青少年的心理健康 AI 陪伴应用，由于业务涉及未成年人隐私数据与心理健康敏感信息，安全架构在设计中遵循“零信任、最小权限、纵深防御”三大原则。系统由多端（学生 / 教师 / 家长 / Web）+ 后端服务（Spring Boot）+ AI 引擎（FastAPI）+ 多数据存储（PostgreSQL / MongoDB / Redis）组成，部署于 Docker + Kubernetes 集群内，由 Nginx 完成统一接入与负载均衡。

### 1.3 安全设计原则

| 原则 | 说明 |
|------|------|
| 零信任 | 不信任任何内外部请求，所有访问均需鉴权与授权 |
| 最小权限 | 角色、密钥、网络策略均遵循最小必要权限 |
| 纵深防御 | 网关层 / 服务层 / 数据层多层防护，单点失陷不可贯穿 |
| 数据加密 | 传输 TLS、存储 AES-256-GCM、密钥环境隔离 |
| 隐私隔离 | 学生对话原文对家长绝对不可见，仅可见情绪趋势摘要 |
| 未成年人保护 | 风险检测自动触发应急预案，家长侧仅推送风险等级 |

## 2. 安全边界划分

### 2.1 客户端层边界

| 边界 | 防护措施 |
|------|----------|
| 设备信任 | 不信任客户端设备，所有接口必须携带有效 JWT |
| 本地存储 | Token 仅存于安全存储（Flutter Secure Storage / Web HttpOnly Cookie） |
| 代码混淆 | Flutter release 构建进行 tree-shaking 与混淆；React 生产构建压缩 |
| 证书校验 | 客户端强制校验服务端 TLS 证书，禁用明文 HTTP |
| 防调试 | Release 包关闭调试日志，禁止 root/越狱设备运行（可选） |

### 2.2 API 网关层边界

| 边界 | 防护措施 |
|------|----------|
| TLS 终结 | Nginx 统一终结 HTTPS（TLS 1.2+），上游 HTTP 内网通信 |
| 路由白名单 | 仅放行已注册路由，未匹配返回 404 |
| 速率限制 | Nginx `limit_req` 按 IP / 用户限流 |
| 请求大小 | 限制 `client_max_body_size`，防止大包攻击 |
| 头部清洗 | 移除 `X-Forwarded-For` 不可信字段，注入真实客户端 IP |
| WAF（可选） | 接入云 WAF 拦截 SQL 注入、XSS、CC 攻击 |

### 2.3 服务层边界

| 边界 | 防护措施 |
|------|----------|
| 认证 | `JwtAuthenticationFilter` 校验 Bearer Token 签名与过期时间 |
| 授权 | `SecurityFilterChain` 基于 RBAC 限制端点访问 |
| 输入校验 | Jakarta Validation 注解（`@Valid`、`@NotBlank`、`@Size` 等） |
| 输出编码 | 统一 `ApiResponse` 包装，避免直接返回实体对象 |
| 异常屏蔽 | `GlobalExceptionHandler` 统一处理，不泄露堆栈信息 |
| 服务间通信 | 后端调用 AI 引擎走内网 HTTP，不暴露公网 |

### 2.4 数据层边界

| 边界 | 防护措施 |
|------|----------|
| 数据库认证 | PostgreSQL / MongoDB / Redis 强密码 + 网络隔离 |
| 网络隔离 | 数据库仅允许集群内 Service 访问，不暴露 NodePort |
| 字段加密 | 敏感字段（对话内容、情绪摘要）使用 AES-256-GCM 加密存储 |
| 备份加密 | 数据库备份文件加密存储于对象存储 |
| 最小权限账号 | 应用账号仅授予 CRUD 权限，不授予 DDL / 管理员权限 |
| 审计日志 | 关键表（绑定关系、应急预案）开启变更日志 |

## 3. 身份认证机制

### 3.1 JWT 认证流程

```
┌─────────┐  1.POST /api/v1/users/login      ┌───────────────┐
│  客户端  │ ─────────────────────────────────►│  后端服务     │
│         │  {username, password}             │  Spring Boot  │
│         │                                   │               │
│         │  2.校验密码 (BCrypt 12 轮)         │               │
│         │  3.生成 JWT (userId,role,iat)      │               │
│         │ ◄─────────────────────────────────│               │
│         │  {token: "Bearer xxx", ...}        │               │
│         │                                   │               │
│         │  4.携带 Authorization: Bearer xxx  │               │
│         │ ─────────────────────────────────►│               │
│         │                                   │  JwtAuthFilter│
│         │                                   │  · 校验签名    │
│         │                                   │  · 校验过期    │
│         │                                   │  · 注入权限    │
│         │  5.业务响应                        │               │
│         │ ◄─────────────────────────────────│               │
└─────────┘                                   └───────────────┘
```

### 3.2 令牌生命周期

| 阶段 | 说明 |
|------|------|
| 签发 | 登录 / 注册成功后由 `JwtUtil.generateToken` 生成 |
| Claims | `userId`、`role`、`iat`（签发时间） |
| 算法 | HMAC-SHA（对称签名，密钥服务端持有） |
| 有效期 | 86400000 ms = 24 小时 |
| 携带方式 | HTTP Header `Authorization: Bearer <token>` |
| 校验 | `JwtAuthenticationFilter` 在每次请求时校验签名与过期时间 |
| 失效 | 过期后服务端拒绝，客户端需重新登录 |

### 3.3 权限注入

| 步骤 | 实现位置 |
|------|----------|
| 解析 Token | `JwtAuthenticationFilter` |
| 提取 role | 从 claims 中读取 `role` |
| 权限拼接 | `"ROLE_" + role.toUpperCase()` |
| 写入上下文 | `SecurityContextHolder.getContext().setAuthentication(...)` |
| 端点校验 | `@PreAuthorize` 或 `SecurityFilterChain` 中 `hasRole(...)` |

### 3.4 刷新策略

当前版本采用单 Token 模型，**未实现 Refresh Token**：

- Token 有效期 24 小时，过期后需重新登录；
- 客户端在 Token 接近过期时（如剩余 1 小时）主动提示用户；
- 改进计划：引入 Refresh Token（有效期 7 天，存于 HttpOnly Cookie），AccessToken 有效期缩短至 1 小时（详见第 10 章）。

## 4. 授权控制机制

### 4.1 RBAC 角色矩阵

| 角色 | 适用对象 | 主要权限范围 |
|------|----------|--------------|
| `STUDENT` | 学生用户 | 心情打卡、AI 对话、查看自身记录、控制家长授权 |
| `TEACHER` | 教师用户 | 查看所辖学生风险概览、危机干预记录 |
| `PARENT` | 家长用户 | 查看孩子情绪趋势、接收风险预警、与大星对话 |
| `ADMIN`（预留） | 平台管理员 | 系统配置、急救资源管理、审计（暂未启用） |

### 4.2 API 端点权限映射表

| 端点 | HTTP 方法 | 允许角色 | 说明 |
|------|-----------|----------|------|
| `/health` | GET | 公开 | 健康检查 |
| `/api/v1/users/register` | POST | 公开 | 学生 / 教师注册 |
| `/api/v1/users/login` | POST | 公开 | 学生 / 教师登录 |
| `/api/v1/parents/register` | POST | 公开 | 家长注册 |
| `/api/v1/parents/login` | POST | 公开 | 家长登录 |
| `/api/v1/risk/crisis/hotlines` | GET | 公开 | 危机热线（无需登录即可访问） |
| `/api/v1/content/**` | GET | 公开 | 静态内容资源 |
| `/ws/**` | GET / WS | 公开（业务层鉴权） | WebSocket 端点 |
| `/api/v1/parents/**` | * | `PARENT` | 家长专属接口（绑定、对话、查看摘要等） |
| `/api/v1/mood/**` | * | `STUDENT`, `TEACHER`, `PARENT` | 心情记录相关 |
| `/api/v1/chat/**` | * | `STUDENT`, `TEACHER`, `PARENT` | AI 对话相关 |

### 4.3 授权执行点

| 层级 | 机制 | 配置位置 |

| 数据级 | 服务层校验资源归属（如学生仅能查自己） | 各 Service 实现 |

## 5. 加密策略

### 5.1 传输加密

| 通道 | 加密方式 |
|------|----------|
| 客户端 ↔ Nginx | TLS 1.2 / 1.3（HTTPS、WSS） |
| Nginx ↔ 后端服务 | 内网 HTTP（K8s Pod 间） |
| 后端 ↔ AI 引擎 | 内网 HTTP（K8s Pod 间） |
| 后端 ↔ 数据库 | 内网连接，PostgreSQL 默认 SSL 可选开启 |
| Redis | 内网访问，开启 AUTH 密码 |

### 5.2 存储加密

| 数据类别 | 加密算法 | 密钥来源 | 说明 |
|----------|----------|----------|------|
| 用户密码 | BCrypt（12 轮） | 每用户盐值 | 单向哈希，不可逆 |
| 对话原文 | AES-256-GCM | `ENCRYPTION_KEY` 环境变量（32 字节） | IV 12 字节随机，GCM Tag 128 位 |
| 情绪摘要 | AES-256-GCM | 同上 | 加密后入库 |
| 家长对话消息 | AES-256-GCM | 同上 | `parent_chat_messages` 表 |
| 用户级密钥派生 | SHA-256(userId) | 派生密钥 | 用于用户数据隔离 |

### 5.3 AES-GCM 加密流程

```
┌──────────────┐    ┌──────────────────────────────────────┐
│  明文消息    │ ─► │  EncryptionUtil.encrypt(plaintext):  │
│ (对话/摘要)  │    │  1. 生成 12 字节随机 IV               │
└──────────────┘    │  2. 初始化 AES/GCM/NoPadding Cipher   │
                    │  3. GCM Tag 长度 128 位               │
                    │  4. 加密得到 ciphertext + tag         │
                    │  5. 返回 Base64(IV || tag || cipher)  │
                    └──────────────┬───────────────────────┘
                                   │
                                   ▼
                    ┌──────────────────────────────────────┐
                    │  数据库存储密文（chat_messages 等）   │
                    └──────────────────────────────────────┘
                                   │
                                   ▼
                    ┌──────────────────────────────────────┐
                    │  EncryptionUtil.decrypt(ciphertext): │
                    │  1. Base64 解码                       │
                    │  2. 拆分 IV / tag / cipher            │
                    │  3. 初始化 Cipher 解密                │
                    │  4. 校验 GCM Tag（完整性 + 真实性）   │
                    │  5. 返回明文                          │
                    └──────────────────────────────────────┘
```

### 5.4 端到端加密

| 流程 | 说明 |
|------|------|
| 学生对话 | 学生端 → 后端 → AI 引擎处理 → 加密入库；家长无法获取对话原文 |
| 家长对话 | 家长端 → 后端 → AI 引擎（独立 System Prompt）→ 加密入库 |
| 隔离机制 | 学生对话与家长对话使用独立表（`chat_messages` vs `parent_chat_messages`），AI 引擎使用不同 System Prompt |

## 6. 输入验证与输出编码

### 6.1 Jakarta Validation 注解

| 注解 | 用途 | 示例位置 |
|------|------|----------|
| `@NotBlank` | 字符串非空 | `ParentLoginRequest.username` |
| `@NotNull` | 对象非 null | `BindStudentRequest.studentId` |
| `@Size` | 长度限制 | 密码最小长度、用户名长度 |
| `@Email` | 邮箱格式 | 注册邮箱字段 |
| `@Pattern` | 正则约束 | 手机号、验证码格式 |
| `@Valid` | 嵌套校验 | Controller 方法参数 |

Controller 层统一使用 `@Valid` 触发校验，校验失败由 `GlobalExceptionHandler` 捕获 `MethodArgumentNotValidException` 并返回 400。

### 6.2 XSS 防护

| 措施 | 实现方式 |
|------|----------|
| 输出编码 | 统一 `ApiResponse` 包装；前端 React 默认转义、Flutter Text 组件不解析 HTML |
| 输入过滤 | 服务层对富文本字段进行 HTML 标签剥离 |
| Content-Type | 接口统一返回 `application/json`，禁止 `text/html` |
| CSP | 前端页面配置 Content-Security-Policy（计划项） |

### 6.3 SQL 注入防护

| 措施 | 实现方式 |
|------|----------|
| ORM | Spring Data JPA / Hibernate 参数化查询 |
| 禁止拼接 | 代码规范禁止字符串拼接 SQL |
| 最小权限 | 数据库账号仅授予 CRUD，无 DDL / 管理员权限 |

## 7. 安全头与 CORS 策略

### 7.1 安全响应头

| 头部 | 当前状态 | 建议 |
|------|----------|------|
| `X-Frame-Options` | 已禁用（frameOptions disabled） | 如前端不嵌入 iframe，建议恢复 DENY |
| `X-Content-Type-Options` | 未显式配置 | 建议设置 `nosniff` |
| `Strict-Transport-Security` | 未显式配置 | 建议设置 `max-age=31536000; includeSubDomains` |
| `Content-Security-Policy` | 未配置 | 前端建议配置白名单 |
| `X-XSS-Protection` | 未配置 | 建议设置 `1; mode=block` |
| `Referrer-Policy` | 未配置 | 建议设置 `strict-origin-when-cross-origin` |

### 7.2 CORS 策略

| 配置项 | 值 |
|--------|----|
| 允许来源 | `localhost`、`*.starisle.com`、`*.starisle.cn` |
| 允许方法 | GET、POST、PUT、DELETE、OPTIONS |
| 允许头 | `Authorization`、`Content-Type` 等 |
| 凭证 | 根据业务需要配置 |
| 配置位置 | `CorsConfig.java` |

> 注：当前 `X-Frame-Options` 已禁用，若前端无 iframe 嵌入需求，应在 `SecurityConfig` 中恢复 `frameOptions().deny()`，以防范点击劫持。

## 8. 速率限制与防护

### 8.1 速率限制策略

| 层级 | 机制 | 限制值（建议） |
|------|------|----------------|
| 网关层 | Nginx `limit_req_zone` | 单 IP 100 req/s，突发 50 |
| 登录接口 | 后端限流 | 单 IP 10 次 / 分钟，失败 5 次锁定 15 分钟 |
| 注册接口 | 后端限流 | 单 IP 5 次 / 小时 |
| AI 对话 | 后端限流 | 单用户 30 条 / 分钟 |
| 风险推送 | WebSocket 心跳 | 单连接 1 req/s |

### 8.2 其他防护

| 防护项 | 说明 |
|--------|------|
| 暴力破解 | 登录失败计数 + 临时锁定 + BCrypt 12 轮提高破解成本 |
| 机器人 | 关键接口接入图形验证码 / 滑块（计划项） |
| DDoS | 云厂商 DDoS 防护 + Nginx 限流 |
| 会话固定 | 登录后重新签发 Token，无服务端 Session |
| 重复提交 | 关键写操作（注册、绑定）使用幂等键或防重 Token |

## 10. 已知安全限制与改进计划

### 10.1 已知安全限制

| 序号 | 限制项 | 风险描述 | 影响等级 |
|------|--------|----------|----------|
| 1 | CSRF 已禁用 | 当前为无状态 API，风险较低；但若未来引入 Cookie 凭证，需重新评估 | 低 |
| 2 | `X-Frame-Options` 已禁用 | 存在点击劫持风险 | 中 |
| 3 | 未实现 Refresh Token | Token 有效期 24 小时偏长，泄露窗口较大 | 中 |
| 4 | JWT 使用 HMAC 对称签名 | 密钥泄露后可任意伪造 Token | 中 |
| 5 | 未配置 HSTS / CSP | 中间人、XSS 防护不足 | 中 |
| 6 | 加密密钥单一 | 全局 `ENCRYPTION_KEY` 共用，未做密钥轮换 | 中 |
| 7 | 速率限制依赖网关 | 后端未实现细粒度限流 | 中 |
| 8 | 审计日志不完整 | 关键操作（绑定、删除、风险推送）审计粒度不足 | 中 |
| 9 | 密钥管理 | 密钥存于环境变量，未使用 KMS / Vault | 中 |
| 10 | 第三方依赖 | 未接入 SCA 工具，依赖漏洞跟踪不足 | 低 |
